# /pr

현재 브랜치의 변경으로 Pull Request를 만든다. 규칙의 단일 출처는 아래이며, 이 커맨드는 그것을
**실행할 뿐 규칙을 다시 정의하지 않는다**:

- 본문 구조·각 섹션이 요구하는 것: [`.github/pull_request_template.md`](../../.github/pull_request_template.md)
- 제목·라벨 규칙, 기준의 배경: [`.claude/docs/pr-checklist.md`](../docs/pr-checklist.md)

> **요구사항**: GitHub CLI(`gh`) 설치 + `gh auth login` 인증. 없으면 초안 작성까지만 가능하고
> 생성 단계에서 실패하므로, 시작 전에 `gh auth status`로 확인한다.

## 절차

### 1. base 결정

인자로 받은 브랜치, 없으면 `main`. `origin/<base>`가 있으면 그것을 기준으로 삼는다.
현재 브랜치가 base면 "PR로 만들 변경이 없다"고 알리고 멈춘다.

### 2. 맥락 수집

- `git log origin/<base>..HEAD --oneline` — **커밋들이 곧 의도다**
- `git diff origin/<base>..HEAD --stat` — 변경 파일
- 커밋 본문과 `.claude` 문서 변경에서 **결정·이탈**을 읽는다

### 3. 본문 작성

`.github/pull_request_template.md`를 읽어 각 섹션을 diff·커밋을 근거로 채운다.

- **"의도적으로 안 한 것 / 후속"과 "검증"(값)은 반드시 채운다** — diff에 담기지 않는 정보다.
  검토했으나 채택하지 않은 것도 여기 적는다.
- diff·커밋·`.claude` 문서에서 읽을 수 있는 것은 중복하지 않는다(의도·제외·검증 결과·이탈 위주).
- **확실치 않으면 추측하지 말고 사용자에게 묻는다.**

### 4. 다이어그램 (mermaid)

아래 중 하나가 변경에 포함되면 본문에 넣는다. 없으면 넣지 않는다 — **그림을 위한 그림은 노이즈다.**

- **여러 레이어를 가로지르는 흐름** — 폴링 → 보간 → 마커 렌더처럼 `entities`·`widgets`·`shared`를
  관통하는 경로 → `sequenceDiagram`
- **상태 기계** — 전이가 3개 이상이고 잘못된 전이가 버그가 되는 것(예: 바텀시트 peek↔full↔닫힘,
  드래그·진입·퇴장) → `stateDiagram-v2`
- **결정론적 분기** — 순서가 곧 규칙인 것(예: 지도 센터 이동 판정) → `flowchart TD`

원칙:

- 정책 문서에 이미 그림이 있으면 **링크**하고, PR에는 리뷰어가 diff 전에 방향을 잡아야 하는
  **핵심 1개만 인라인**한다. 다이어그램의 단일 출처는 정책 문서다.
- 레이아웃은 **세로**(`flowchart TD`)로 그린다 — PR 컬럼에서 가로 스크롤이 생기지 않게.
- **넣지 않는 것**: FSD 레이어 구조도(`.claude/rules/architecture/fsd.md`에 이미 있음),
  파일 나열형 그림 — diff와 중복이다.

### 5. 제목

지배적 변경으로 type을 정하고, 커밋과 동일한 Conventional Commits + 한국어 요약으로 쓴다.
**squash 병합 시 이 제목이 그대로 `main`의 커밋 메시지가 된다.**

### 6. 라벨

변경 유형은 본문이 아니라 **라벨**로 표시한다. 커밋 prefix와 라벨은 **이름이 다르므로**
[`pr-checklist.md`의 매핑표](../docs/pr-checklist.md#커밋-prefix--github-라벨)를 따른다
(`feat` → `feature`, `docs` → `documentation` 등).

### 7. 검증 — 실제로 돌리고 값으로 적는다

통과 여부만 쓰지 않는다. **체크박스는 안 돌려도 찍을 수 있지만 값은 돌려야 나온다.**

```bash
pnpm --filter web lint
pnpm --filter web build   # tsc -b + vite
pnpm --filter web test
```

- 테스트는 **건수**로 적는다(예: `82 passed`).
- 번들이 바뀌었으면 `pnpm build` 출력의 **gzip 실측치**를 적는다.
- **성능·크기를 주장하려면 측정치를 낸다.** "부드러워졌다"·"가벼워졌다" 같은 근거 없는 서술은
  쓰지 않는다. 재보지 않았다면 재보지 않았다고 적는다(→ "한계 / 미검증 / 비고").
- 못 돌렸으면 그 이유를 적는다.

### 8. 선(先)검증

버스 API·데이터에 대한 가정이 있으면 [`.claude/scripts/probe-bus-api.sh`](../scripts/)로
**실제로 호출해** 확인하고, 응답 값을 그대로 인용한다. **추측 금지.**
(전례: 명세의 `tmX/tmY`가 실제 응답에서는 `null`이었고 좌표는 `gpsX/gpsY`였다.)

### 9. 스크린샷 / 데모

사용자에게 보이는 동작이 바뀐 PR이면 첨부한다. UI·동작 변경이 없으면(문서·리팩터링·내부 정리)
건너뛴다 — **보여줄 게 없는데 넣는 그림은 노이즈다.**

**jsdom에는 레이아웃·페인트가 없어 애니메이션 연출·드래그 제스처·프레임률은 자동 테스트로
검증되지 않는다.** 이런 변경일수록 화면이 유일한 증거이므로 반드시 첨부한다.

절차:

1. **캡처** — 개발 서버를 띄우고(`pnpm --filter web dev`) 브라우저 도구로 화면을 잡는다.
   - 정지 화면: 스크린샷을 파일로 저장한다.
   - 움직임(연출·드래그·전환): **GIF로 녹화**한다. 정지 화면으로는 애니메이션을 보여줄 수 없다.
   - 동작이 바뀐 PR은 **Before / After를 나란히** 붙인다. 하나만 있으면 무엇이 달라졌는지 모른다.
2. **업로드** — `/uploading-attachments <파일경로>` 스킬로 GitHub CDN URL을 받는다.
   **이미지를 저장소에 커밋하지 않는다.**
3. **본문에 삽입** — 받은 URL을 "스크린샷 / 데모" 섹션에 넣는다.

> **경고**: 업로드에 쓰는 GitHub API는 비공개이며, **올린 이미지는 CDN에서 삭제할 수 없다.**
> 화면에 개인정보·토큰·실제 좌표 등 민감한 값이 찍히지 않았는지 올리기 전에 확인하고,
> 사용자에게도 이 사실을 알린 뒤 승인받는다.

### 10. 확인 후 생성

초안(제목·본문·라벨)을 사용자에게 보여주고 **승인받은 뒤에만** 실행한다.

```bash
git push -u origin <branch>          # 원격에 없을 때만. 승인 후
gh pr create --base <base> --title "…" --body "…" --assignee @me
gh pr edit --add-label <label>
```

담당자는 `@me`(현재 인증 사용자)로 배정한다 — **계정명을 하드코딩하지 않는다.**

## 규칙

- **push·PR 생성·라벨 부여·이미지 업로드 등 외부에 나가는 작업은 반드시 먼저 확인받는다.**
  승인 전엔 실행하지 않는다. 특히 이미지는 **올리면 지울 수 없다**(9단계 경고).
- 저장소 식별자(GitHub 계정·URL)를 본문에 하드코딩하지 않는다.
- PR 본문에 `- [ ]` 체크리스트를 만들지 않는다. 템플릿에 없는 구조를 지어내지 말 것.
  검증은 체크가 아니라 **값**으로 적는다(7단계).
